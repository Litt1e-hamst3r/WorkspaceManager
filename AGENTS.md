# Workspace Manager Agent Guide

本文件定义仓库内的默认协作约束，作用域为整个项目树。

## 项目目标
- 本项目是一个 Windows 桌面管理工具
- 当前阶段优先产出可开发、可验证的 MVP
- 核心能力围绕托盘常驻、桌面图标控制、布局保存/恢复、桌面整理、模式切换

## 当前技术方向
- 首选技术栈：`C#` + `.NET 8` + `WPF`
- Windows 交互优先通过独立 `Interop` 层封装
- 配置与快照优先使用 `JSON` 持久化
- 如无明确需求，不要引入 `Electron`、`Python GUI` 或跨平台改造

## 推荐目录结构
- `docs/`：产品、技术、设计、任务文档
- `src/App/`：应用启动与依赖注入
- `src/UI/`：WPF 页面、窗口、ViewModel
- `src/Application/`：用例编排与应用服务
- `src/Domain/`：领域模型与规则对象
- `src/Infrastructure/`：配置、日志、文件系统实现
- `src/Interop/`：Win32 / Shell / PInvoke 封装
- `tests/`：单元测试与集成测试

## 开发原则
- 优先修根因，不做表面补丁
- 优先小步增量，不一次性铺太大框架
- 新增模块必须围绕 MVP 目标，避免无关扩展
- 非必要不要引入额外第三方依赖
- 涉及 Windows API 的实现要和业务逻辑隔离

## 代码风格
- 保持命名清晰，避免单字母命名
- 优先使用小而明确的服务类
- 接口命名使用 `I` 前缀，类名使用完整业务语义
- 不要随意添加行内注释，优先通过命名表达意图
- 配置对象、领域对象、服务对象分层清晰

## 架构约束
- `UI` 层不直接调用 Win32 细节；通过 `Application` 或 `Interop` 抽象访问
- `Domain` 层不依赖具体 UI 或文件系统实现
- `Infrastructure` 不承载业务编排逻辑
- `Interop` 层负责平台相关实现，但不要夹带产品规则逻辑

## 文档要求
- 影响功能边界、模块划分、存储结构的变更，要同步更新 `docs/`
- 新增重要模块时，优先补最小设计说明或 README
- 需求变更优先更新 `docs/PRD_v1.md`
- 技术架构调整优先更新 `docs/Technical_Design_v1.md`

## 测试要求
- 新增业务逻辑时，优先补对应单元测试
- 新增规则引擎、模式编排、配置解析相关逻辑时，应有测试覆盖
- 不要为了通过测试而改动与当前任务无关的行为
- 若仓库尚未建立测试工程，可在搭建时按 `tests/` 结构补齐

## 当前优先级
- 第一优先：托盘、桌面图标显示/隐藏、布局保存/恢复
- 第二优先：模式切换、手动整理、规则配置
- 第三优先：自动整理、撤销、演示模式增强

## 提交前检查
- 只修改与当前任务直接相关的文件
- 确认文档、命名、目录结构一致
- 若已有可运行测试，优先执行受影响范围内的测试
- 若未执行测试，在最终说明中明确指出
